4. 系统总体架构

建议采用 “目录树 + 多智能体 + 知识检索 + 反馈学习” 四层架构：

4.1 目录层

负责：

标准章节树

文件归类

节点映射

节点状态管理

4.2 智能体层

负责：

按节点执行解析、检索、审评、质控、汇总

4.3 知识层

负责：

本地知识库检索

法规/指导原则/药典/历史经验召回

证据片段返回

4.4 学习层

负责：

记录反馈

比较结果

自动提示优化

自动工作流优化

经验沉淀与复用

5. 多智能体设计

建议至少包含以下 8 类智能体。

5.1 目录解析智能体（Parser Agent）
职责

解析上传资料

识别目录编号与标题

归并到合法 section_id

生成节点-文件映射

输入

原始文件

标准章节树

输出
{
  "file_id": "xxx",
  "raw_detected_section_id": "3.2.S.2.2.3",
  "mapped_section_id": "3.2.S.2.2",
  "confidence": 0.94
}
提示词核心

你是 CTD Module 3 目录解析器。

你的任务是识别文件实际所属章节，并映射到系统允许的最近合法节点。

不允许生成不存在的章节。

若识别到超层级章节，必须归并到最近合法父节点。

5.2 节点摘要智能体（Section Summarizer Agent）
职责

对节点下全部文件进行结构化摘要

提取章节关键事实

输出

工艺路线

关键参数

标准项目

方法学验证摘要

稳定性结论

风险点初筛

提示词核心

仅提取当前 section_id 相关信息

不跨节自由推断

输出必须结构化

明确区分“资料明确写明”“合理推断”“资料缺失”

5.3 知识检索智能体（Knowledge Retriever Agent）
职责

从本地知识库按节点检索证据：

适用指导原则

法律法规

中心制度

药典要求

历史同类缺陷/反馈

既往审评经验

检索输入

section_id

节点摘要

产品信息

注册分类

剂型/原料药属性

输出
{
  "section_id": "3.2.P.5.2",
  "retrieved_rules": [],
  "pharmacopoeia_refs": [],
  "historical_patterns": []
}
提示词核心

按当前节点召回最相关规则，不返回泛泛法规

优先返回可操作审评依据

返回证据时必须包含出处、适用范围、触发条件

5.4 节点审评智能体（Section Reviewer Agent）
职责

按节点完成核心审评。

输入

section_id

节点摘要

章节关注点

检索到的法规与经验

上游节点上下文

输出
{
  "section_id": "3.2.P.3.3",
  "review_status": "reviewed_with_issues",
  "summary": "...",
  "key_findings": [],
  "risks": [],
  "deficiencies": [],
  "questions_to_applicant": []
}
提示词核心

你是化学药药学审评员，只审当前章节

审评必须基于：节点资料 + 节点关注点 + 本地知识证据

不得脱离当前节点泛化结论

缺陷必须写明：问题点、依据、风险、建议补充内容

若证据不足，必须输出“证据不足”，不得猜测

5.5 交叉一致性智能体（Consistency Agent）
职责

检查跨章节一致性，例如：

3.2.S.2.2 工艺 vs 3.2.S.2.4 关键步骤控制

3.2.P.2 开发 vs 3.2.P.3 生产工艺

3.2.P.5.1 质量标准 vs 3.2.P.5.2 方法 vs 3.2.P.8.3 稳定性

输出

矛盾点

断链点

缺失桥接点

提示词核心

只检查跨节点一致性

不复述单节点结论

输出必须指向具体 section_id

5.6 审评汇总智能体（Lead Reviewer Agent）
职责

汇总全案结论：

原料药总体结论

制剂总体结论

总体风险图谱

关键发补问题清单

提示词核心

汇总不能覆盖节点事实

必须保留各节点主要证据链

输出总评时要区分关键缺陷与一般缺陷

5.7 质量控制智能体（QA Agent）
职责

对智能体输出做二次检查：

结论是否有证据

是否超出资料内容

是否遗漏高风险问题

是否存在模板化空话

提示词核心

你不重新审评，只做质量复核

逐条验证“结论-证据”绑定关系

发现无依据结论必须打回

5.8 学习优化智能体（Optimizer / Self-Evolution Agent）
职责

根据反馈持续优化：

提示词

检索策略

路由策略

工作流结构

经验模板

这是系统最关键的“自我进化”模块。

6. 自我进化设计（重点）

建议采用 “经验记忆 + 反思优化 + 自动提示优化 + 工作流搜索优化” 的组合机制。

6.1 可利用的反馈类型

系统必须把以下反馈变成可学习信号：

人工反馈

审评员采纳/驳回

修改意见

标注误判/漏判

监管反馈

实际发补问题

企业回复后结论变化

最终审评意见

流程反馈

哪些节点经常出错

哪些规则命中率低

哪些提示词导致冗长/空泛/错判

结果反馈

缺陷准确率

漏报率

引用证据覆盖率

审评员满意度

节点级 precision / recall

6.2 三层学习机制
第一层：案例经验记忆（Case Memory）

保存每次任务的：

产品类型

章节

发现的问题

最终是否被人工确认

对应证据

最终反馈

用途：

下一次遇到相似产品/剂型/章节时优先召回

这和 Reflexion 中“把反馈转成可复用语言记忆”的思想一致：不是改模型权重，而是把有效反思写入持久经验，供下次决策使用。

第二层：提示词自动优化（Prompt Optimization）

对每个节点审评提示词维护版本库：

基础提示词

历史优化版

当前生产版

每次根据反馈自动生成：

哪句提示导致误判

哪类规则表达不清

哪类约束需要更明确

再用自动提示优化方法改写提示词并离线评估。APO 的核心思路就是用数据对提示进行自动批评、编辑和搜索优化；DSPy 也把“以指标驱动优化模块提示/程序”作为正式机制。

建议实现：

Prompt Critic：根据失败案例产出“提示缺陷描述”

Prompt Rewriter：生成候选提示词

Prompt Evaluator：在验证集上比对

Prompt Registry：只发布指标更优版本

第三层：工作流自动优化（Workflow Optimization）

不仅优化单条提示，还优化多智能体流程本身：

可调整项包括：

是否先摘要后审评

哪些节点先做一致性检查

哪些节点需要强制检索药典

哪些节点需要二次复核

哪些产品类型启用特殊子流程

这类“把工作流视为可搜索对象”的方法，与 AFlow 这类自动工作流优化框架的思想一致：根据执行反馈搜索更好的代理协作图。

6.3 自我进化闭环

系统必须实现如下闭环：

Step 1 运行

执行一次完整审评任务。

Step 2 收集反馈

采集：

人工修订

被驳回结论

最终发补

审评员评分

企业回复后的反证

Step 3 失败归因

由优化智能体分析失败原因，分类为：

解析错误

检索遗漏

提示不清

路由错误

规则缺失

工作流顺序不合理

历史经验未命中

Step 4 生成优化候选

分别对：

Prompt

检索 query 模板

规则匹配逻辑

agent 编排顺序
生成候选版本。

Step 5 离线验证

在历史案件验证集上跑 A/B 测试，比较：

缺陷命中率

漏报率

审评员采纳率

证据引用完整率

平均 token / 时间成本

Step 6 发布

仅发布优于当前版本的候选。

Step 7 沉淀经验

将本次有效经验写入经验库：

{
  "task_type": "化学药-原料药-3.2.S.2.4",
  "pattern": "关键步骤设定仅给结论、无开发依据",
  "evidence_rule": "需联动3.2.S.2.6工艺开发与3.2.S.2.5验证资料判断",
  "best_prompt_patch": "...",
  "approved_by_human": true
}

→ QA校验
→ 存储结果
8. 提示词编写规范

所有智能体提示词建议拆成 4 段：

8.1 Role

定义身份
例如：

你是化学药原料药生产工艺审评员

你是 CTD 章节解析器

你是审评结果质控员

8.2 Scope

限定范围
例如：

你只审当前 section_id

不得跨章节自行补充事实

只能依据给定资料和知识证据判断

8.3 Method

规定步骤
例如：

先提取关键事实

再对照关注点

再对照法规/药典/历史经验

最后输出问题与结论

8.4 Output Contract

规定 JSON 输出格式
禁止自由发挥文本

9. 推荐的核心提示词模板
9.1 节点审评智能体 Prompt 模板
你是化学药药学审评智能体，当前只审评一个章节。

【当前章节】
section_id: {section_id}
section_name: {section_name}

【任务】
1. 仅基于当前章节资料、章节关注点、知识库证据进行审评
2. 判断该章节资料是否充分、是否存在风险、是否需要发补
3. 输出结构化结论，不得输出空泛套话
4. 若证据不足，明确写“证据不足”，不得猜测

【章节关注点】
{focus_points}

【节点摘要】
{section_summary}

【知识库证据】
{knowledge_evidence}

【上游上下文】
{upstream_context}

【输出要求】
返回JSON，字段必须包括：
- section_id
- review_status
- summary
- key_findings
- risks
- deficiencies
- questions_to_applicant
- evidence_refs

【缺陷编写规则】
每条缺陷必须包括：
- issue
- basis
- risk
- requested_action

【禁止事项】
- 不得引用未给出的法规
- 不得把其他章节内容当作本章节已证明事实
- 不得只说“建议完善”而不说明具体补充内容
9.2 反馈反思智能体 Prompt 模板
你是审评系统优化智能体。

【目标】
根据本次任务结果与人工反馈，分析失败原因并提出可执行优化方案。

【输入】
- 原始提示词
- 原始输出
- 人工修订结果
- 最终采纳结果
- 命中/漏报统计
- 相关案例经验

【任务】
1. 判断问题属于：解析错误 / 检索遗漏 / 提示不清 / 路由错误 / 规则缺失 / 工作流问题
2. 产出：
   - failure_analysis
   - prompt_patch
   - retrieval_patch
   - workflow_patch
   - reusable_lesson
3. 优先给出最小改动、最大收益的优化建议
4. 输出应适合进入下一轮自动评测

【输出格式】
JSON:
- failure_type
- root_cause
- prompt_patch
- retrieval_patch
- workflow_patch
- reusable_lesson
- confidence
10. 经验沉淀机制

系统必须建立两个经验库：

10.1 节点经验库

按 section_id 存储常见问题：

3.2.S.2.4 常见问题：关键工艺参数只有范围无依据

3.2.P.5.2 常见问题：有关物质方法缺少专属性/区分力

3.2.P.8.3 常见问题：稳定性趋势分析不足、效期支持不足

10.2 产品类型经验库

按产品类型/剂型存储模式：

注射剂

缓控释制剂

复方制剂

无菌原料药

光敏/湿敏产品

下一次任务进入时先做经验召回，再决定提示词和工作流分支。
